最適な手法を選択するのに役立つヒントを次に示します。

-   [デバッガーからのリモート デバッグ](remote-debugging-through-the-debugger.md)最適な方法では、通常します。 単に 1 つのサーバーがあるし、1 つのクライアントとこれらが相互に接続できる自由に、クライアントとサーバーの両方で同じデバッガー バイナリがインストールされているし、クライアントが動作するデバッグ技術者との対話を部屋にだれかができます。このサーバーでは、これは推奨される方法です。

    クライアントとサーバーに Windows の任意のバージョンを実行できます。 他の同じバージョンを実行しているはありません。

    デバッガーで使用して、リモート デバッグを使用するには、クライアントがない場合、サーバーに接続要求を送信できるように、サーバーがクライアントに要求を送信できるように、*接続を取り消す*を使用して、 **clicon**パラメーター。

-   [Remote.exe を通じてリモート デバッグ](remote-debugging-through-remote-exe.md)コマンド プロンプト ウィンドウをリモートで制御するために使用します。 KD、CDB、NTSD をリモートで制御するために使用します。 これは、WinDbg で使用できません。

    クライアントには、デバッガー バイナリのコピーがあるない、remote.exe メソッドを使用する必要があります。

-   **プロセス サーバー**または**KD 接続サーバー**デバッグ技術者が、サーバー ルームのメンバーに対して実施することはない場合に使用できます。 実際のすべてのデバッグ作業は、クライアントによって行われます (と呼ばれる、*スマート クライアント*);、サーバー自体に存在する 2 つ目のユーザーが削除されます。

    プロセス サーバーがユーザー モードのデバッグに使用します。KD 接続のサーバーは、カーネル モードのデバッグに使用されます。 このような区別以外に近い方法で動作します。

    このメソッドは、サーバーを実行するコンピューターは、負荷の高い処理負荷を処理できない場合に役立ちます。 また、クライアントを実行している技術者は、シンボル ファイルにアクセスするか、ソース ファイルまたは機密情報とすることはできません、サーバーからアクセスします。 ただし、このメソッドは、その高速またはデバッガーからのリモート デバッグと効率的なありません。 このメソッドは、ダンプ ファイルのデバッグに使用できません。

    参照してください[プロセス サーバー (ユーザー モード)](process-servers--user-mode-.md)と[KD 接続サーバー (カーネル モード)](kd-connection-servers--kernel-mode-.md)詳細についてはします。

-   **Repeater** 2 台のコンピューター間でデータを中継する軽量のプロキシ サーバーします。 デバッガーからのリモート デバッグを実行している場合、またはプロセス サーバーを使用している場合は、クライアントとサーバー間の repeater を追加できます。

    Repeater を使用する必要があります、クライアントとサーバーは、互いに直接通信できませんが、各外部のコンピューターにアクセスできます。 リピータ同様で、リバース接続を使用できます。 行で、2 つのリピータを使用することもが、必要なことはほとんどありません。
