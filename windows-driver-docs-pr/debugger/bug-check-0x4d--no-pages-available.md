# <a name="bug-check-0x4d-nopagesavailable"></a>バグ チェック 0x4D:いいえ\_ページ\_利用可能


いいえ、\_ページ\_使用可能なバグ チェックが 0x0000004D の値を持ちます。 これは、空きページが継続的に使用しないことを示します。

> [!IMPORTANT]
> このトピックはプログラマーを対象としています。 コンピューターを使用しているときに、エラー コードがブルー スクリーンが受信した顧客の場合を参照してください。[トラブルシューティング ブルー スクリーン エラー](https://windows.microsoft.com/windows-10/troubleshoot-blue-screen-errors)します。


## <a name="nopagesavailable-parameters"></a>いいえ\_ページ\_使用可能なパラメーター

|パラメーター|説明|
|--- |--- |
|1|ダーティ ページの合計数|
|2|ページファイル宛てのダーティ ページの数|
|3|バグ チェックの発生時に使用可能な非ページ プールのサイズ|
|4|最新の変更の書き込みエラーの状態。|


<a name="cause"></a>原因
-----

全般的なメモリの統計情報を表示するには、使用、 [ **! vm 3** ](-vm.md)拡張機能。

このバグ チェックの次の理由により発生します。

-   デッドロック、変更、または割り当てられたページ ライターでは、ドライバーがブロックされています。 この例には、ミュー テックスのデッドロックやファイル システム ドライバーまたはフィルター ドライバーのメモリをページアウトへのアクセスが含まれます。 これは、ドライバーのバグを示します。

    パラメーター 1 または 2 のパラメーターが大きい場合、この可能性です。 使用[ **! vm 3**](-vm.md)します。

-   記憶装置ドライバーが要求を処理していません。 この例は、孤立したキューとドライブの応答のないです。 これは、ドライバーのバグを示します。

    パラメーター 1 または 2 のパラメーターが大きい場合、この可能性です。 使用[ **! vm 8**](-vm.md)、その後に[ **! 0 7 処理**](-process.md)します。

-   優先度の高いリアルタイム スレッドが、ワーキング セットからトリミング ページからマネージャーの分散の設定の不足または、変更されたページのライターに書き込むから不足しています。これは、このスレッドを作成するコンポーネントのバグを示します。

    このような状況では、分析が困難です。 使用してお試しください[ **! 準備**](-ready.md)します。 お試しください[ **! 0 7 を処理**](-process.md)をすべてのスレッドの一覧を表示し、過剰なカーネル時間を累積いずれかのかどうかを参照してください。 現在の優先順位とは何もします。 このようなプロセスは、ページを使用できるようにすることからのメモリ管理スレッドがブロックされる可能性があります。

-  十分なプールは、記憶域スタックが変更されたページを書き込むに使用できます。 これは、ドライバーのバグを示します。

    パラメーター 3 が小さい場合は、この可能性です。 使用[ **! vm** ](-vm.md)と[ **! poolused 2**](-poolused.md)します。

問題が見つからない場合は、当初から、接続されているカーネル デバッガーを起動してみてくださいし、状況を監視します。
