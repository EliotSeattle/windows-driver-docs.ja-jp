# <a name="global-navigation-satellite-system-gnss-driver-requirements"></a>グローバルナビゲーションサテライトシステム (GNSS) ドライバーの要件

Windows 10 用のグローバルナビゲーションサテライトシステム (GNSS) ドライバーを開発する際に考慮すべき要件、前提条件、および制約について説明します。

## <a name="general-requirements"></a>全般的な要件

- **ドライバーフレームワーク:** GNSS ドライバーは、未加工の WDM ドライバーや KMDF ドライバーとは対照的に、このインターフェイス定義に基づいて、UMDF 2.0 ドライバー a として記述する必要があります。 UMDF 1.0 ドライバーはサポートされていません。 GNSS driver インターフェイス定義または Microsoft high level オペレーティングシステム (HLOS) の gnss コンポーネント (GNSS アダプターなど) では、WDF、KMDF GNSS ドライバー、および UMDF 2.0 ドライバーは区別されません。ただし、ドライバーがこれに対して必要な機能を提供している場合に限ります。インターフェイスのデザイン。 UMDF 2.0 では、ユーザーモードでしか提供されない機能を必要とする機能を実装するための、より高い安定性、簡潔性、および柔軟性が提供されます。 一般的な規則として、以前のフレームワークがプラットフォームで利用可能な場合、Ihv は、UMDF 2.0 を KMDF に優先する必要があります。

    > [!NOTE]
    >  UMDF 2.0 はすべてのプラットフォームで使用でき、Ihv はユーザーモードで記述されたドライバーを使用することを強くお勧めします。

- **複数のアプリケーションセッション:** アプリケーションセッションは、GNSS ドライバーと直接やり取りする HLOS コンポーネントからの配置セッションです。 GNSS ドライバーは、その状態変数と機能をアプリケーションごとにパーティション分割することによって、複数のアプリケーションセッションをネイティブにサポートするように選択できます。 これはドライバーのオプション機能であり、明確に定義された GNSS ドライバーの機能情報によって示されます。 このオプションの動作をサポートするために、GNSS ドライバーは、HLOS アプリケーションが**CreateFile**中に取得したファイルハンドルを追跡し、それ以降のすべての hlos 操作をアプリケーションセッション固有のファイルハンドルに関連付けている必要があります。 GNSS ドライバーからのこのネイティブサポートにより、HLOS コンポーネントはより柔軟で、さらに多くのプラットフォームにドライバーを公開することができます。 この機能をサポートする GNSS ドライバーは、個々のアプリケーションセッションの状態情報を論理的に分割し、維持する必要がある場合があります。 この機能をサポートしていない GNSS ドライバーでは、論理的なアプリ固有のパーティションではなく、すべてのアプリケーションセッションのグローバル状態を維持するだけで済みます。 後者のモードでは、GNSS ドライバーは複数の並列アプリケーションセッションの存在を無関係し、HLOS からのすべての要求を同じアプリケーションセッションから送信されたものとして扱います。

    ![gnss ドライバーによる複数のアプリケーションセッションのサポート](images/gnss-architecture-1.png)

    複数のアプリケーションセッション用の GNSS ドライバーでのサポートには、HLOS のテストアプリケーションが gnss アダプターと同時に GNSS ドライバーと直接対話できるようにするという利点があります。 テストアプリケーションと GNSS アダプターは、単一の GNSS ドライバーの異なるセッションに同時に要求できるさまざまなアプリケーションと見なされます。 複数のアプリケーションセッションがサポートされていない場合は、GNSS ドライバーを OS ロケーションプラットフォームを通じてでテストする必要があります。そうでない場合は、OS ロケーションプラットフォームをホストしているサービスを停止して、テストアプリケーションに干渉しないようにする必要があります。

- **セッションの修正:** 基になるドライバー (単一ショットまたは追跡) から位置情報を取得する行為は、修正セッションの概念に抽象化されています。 ドライバーは、サポートされている各セッションの種類の少なくとも1つの修正セッションをサポートする必要があります。 セッションの種類は、 [**Gnss\_FIXSESSIONTYPE**](https://docs.microsoft.com/windows-hardware/drivers/ddi/gnssdriver/ne-gnssdriver-gnss_fixsessiontype)列挙型に定義されています。

  - 少なくとも、GNSS のドライバーでは、ワンショット修正セッションがサポートしている必要があります。

  - 距離ベースのトラッキング修正セッションをサポートするドライバーは、シングルショット修正セッションと、距離ベースのトラッキング修正セッションを同時にサポートする必要があります。

  - 時間ベースのトラッキング修正セッションをサポートするドライバーは、単一ショット修正セッションと時間ベースの追跡修正セッションを同時にサポートする必要があります。

  - 距離ベースの追跡修正セッションと時間ベースの追跡修正セッションをサポートするドライバーは、単一ショット修正セッション、距離ベースのトラッキング修正セッション、および時間ベースの追跡修正セッションをサポートする必要があります。

    ![複数の同時修正セッションのドライバーサポート](images/gnss-architecture-2.png)

    ドライバーが複数の同時修正セッションをサポートしているかどうかは、適切に定義されたドライバー機能パラメーターによって表されます。 ドライバーが複数の並列修正セッションを明示的にサポートしていない場合、同じ修正の種類のセッションが既に開始され、アクティブになっていると、新しい修正セッション要求を失敗させる必要があります。 複数の修正セッションのサポートが存在しない場合、負担は HLOS コンポーネント上にあります。これにより、高レベルのアプリケーションからの複数の GNSS 要求が多重化され、GNSS ドライバーへの単一の修正セッション要求にマップされます。

    GNSS ドライバーは、同じ種類の複数の並列修正セッションをサポートするためには必要ありません。 Windows 10 の HLOS GNSS アダプターは、GNSS ドライバーの機能を利用して同じ種類の複数の修正セッションを使用することをサポートしていません。そのため、この機能については、Ihv がこの機能に投資することをお勧めしません。 今後のリリースでは、GNSS アダプターが、ロケーションプラットフォームの上位層から取得されたすべての修正要求に対して GNSS ドライバーによるセッションを直接開始できるようにすることを検討します。これは、セッションを多重化することではありません。 GNSS アダプターでの修正セッションの多重化のサポートにより、ドライバーの実装が簡単になります。これは、アプリケーションや多重化の実装で同じ種類の複数のセッションを処理する必要がないため、ドライバーのメモリ使用量を削減し、gnss ドライバーでサポートされている数より多くの修正セッションを開始する HLOS のケースを処理するために、GNSS アダプターは必要ありません。 デバイスとドライバーが異なる場合は、異なる数の同時修正セッションがサポートされるため、このケースを処理する必要があります。これにより、すべてのケースを処理するために、GNSS アダプターの複雑さが増します。 そのため、Windows 10 では、GNSS アダプターはサポートされている各種類の単一の修正セッションのみを実装し、この機能が必要になるまでドライバーによる複数修正セッションのサポートを無視します。

    各修正セッションはステートフルであり、この明確に定義されたシーケンスに従う必要があります。

    1. 修正セッションを開始しています

    1. 1つまたは複数の修正プログラムを取得する

    1. 必要に応じて修正セッションを変更する

        > [!NOTE]
        > これは少なくとも、GNSS アダプターが同じ型の修正セッションの多重化を処理するまで必要であり、また、GNSS ドライバーでサポートされている数よりも多くの同時修正セッションがアクティブであるケースを処理するために後で必要になる場合もあります。

    1. 修正セッションを停止しています

    修正セッションは、修正セッション ID によって一意に識別されます。 HLOS は、修正セッションのコンテキスト外で位置情報を取得することはできません。 GNSS ドライバーでは、HLOS コンポーネントによる多重化操作を容易にするために、セッションパラメーターを即座に変更できるようにする必要があります。修正セッションを再起動する必要はありません。

    ![gnss ドライバーとアプリケーションの間のレポート通信を修正する](images/gnss-architecture-3.png)

- **修正の種類:** GNSS ドライバーは、基本的な単一ショット修正をサポートする必要があります。 さらに、ドライバーは、高度な修正の種類 (追跡など) をネイティブにサポートする必要があります。 前述のように、追加の修正の種類をサポートすることは、ドライバーが同じ種類の複数の修正セッションをサポートしていない場合でも、サポートされている修正プログラムの種類の少なくとも1つの修正セッションを同時にサポートする必要があることを意味します。 HLOS コンポーネントでは、異なる修正の種類が1つの型に多重化されません。

- **デバイスインターフェイスと PnP:** GNSS ドライバーは、 **WdfDeviceCreateDeviceInterface** API を使用して Microsoft が定義したデバイスインターフェイスを公開する必要があります。これにより、hlos は、gnss ドライバーの到着と出発に関する通知を受け取ることができます。 これは、プラグアンドプレイ (PnP) 設定で GNSS ドライバーを処理する必要があります。また、ドライバーが UMDF 2.0 ドライバーの場合、ユーザーレベルのサービスのクラッシュによって予期しないドライバーのアンロードが発生する場合にも必要になります。 GNSS ドライバーでは、その前ではなく、その下にあるハードウェアが HLOS IOCTL 呼び出しをサポートできる場合にのみ、デバイスインターフェイスが提供されるようにする必要があります。

- **デバイスの電源ポリシー:** GNSS ドライバーは、デバイスの電源ポリシーを管理する必要があり、OS によって生成される電源管理イベントを処理する必要があります。 ドライバーは、 *WDF\_PNPPOWER\_イベント\_コールバックに登録する必要があります。EvtDeviceD0Entry* callback (システムが D0 状態になると WDF によって発生) および*WDF\_PNPPOWER\_イベント\_コールバック。EvtDeviceD0Exit* callback (システムが D0 状態から終了したときに WDF によって発生する)。 GNSS ドライバーは、必要に応じて電源管理を無効にするように構成する必要があります。

    異なるシステム電源の状態にある GNSS デバイスで実行する必要のある正確な電源管理は、実際のオフロード操作があるかどうかにかかわらず、GNSS デバイスの機能に従って調整する必要があります (オフロード操作はサポートされています)。アクティブで、システムと GNSS デバイス間の通信がどのように行われるかを説明します。 一般的に、次のような予測があります。
    - GNSS デバイスは、システムの電源状態に関係なく、アクティブなセッションまたはオフロード操作がない場合に、可能な限り低い電源モードで動作します。
    - オフロードシナリオの場合は、システムの電源状態に関係なく、GNSS デバイスでは、異なる間隔で位置を確認するか、通知を受信する必要があります。そのため、接続されたスタンバイ中でも、GNSS デバイスは D0 状態のままである必要があります (これは画面オフのスリープ状態) でも、ハードウェアは電力消費量を最小値に減らす必要があります。 このモデルは、DMA (ダイレクトメモリアクセス) を使用するデバイス、または UART 上のシリアルポートを使用してホストと通信するために機能します。たとえば、のようになります。 しかし、USB バス経由で接続されている GNSS デバイスでは問題になります。この場合、デバイスの USB 機能は、コネクトスタンバイ中に、デバイスの電源状態が D2 (中断) されている必要があります。 一般的に、USB 経由で接続されているデバイスは、修正セッションまたはオフロード操作が進行中であり、USB バスインターフェイスが中断状態になった後に、低電力の D2 (中断) 状態になる必要があります。 すべてのスリープおよびスリープ解除の電源切り替えは、USB バス経由で通知される必要があります。 GNSS デバイスでアクティブまたはオフロード操作のセッションが修正されている場合、デバイスは帯域内、USB 再開信号を使用して、接続されたスタンバイから SoC またはコアシリコンをウェイクアップできる必要があります。 SoC またはコアシリコンは、GNSS デバイスからの帯域内の USB 再開信号に応答して、最も低い電源状態から復帰できる必要があります。
    - コネクトスタンバイをサポートしていないデバイスは、デバイスが最新のスタンバイまたは休止状態になった時点で、すべてのオフロード操作が取り消されます。 これには、ジオフェンスオフロード、距離追跡、定期的な追跡セッションが含まれます。
    - コネクトスタンバイをサポートしているデバイスは、デバイスがコネクトスタンバイ状態になったときに、すべてのオフロード操作をアクティブのままにし、GNSS デバイスは追跡操作を可能な限り効率的に続行することが期待されます。ジオフェンストリガー条件または追跡セッションの更新が関係している場合の HLOS への通知。 コネクトスタンバイをサポートしているデバイスにオフロード操作がない場合、GNSS デバイスは、可能な限り低い電源状態に移行しますが、HLOS からの場所のセッション要求をリッスンすることができます。 SUPL をサポートするデバイスでは、接続されたスタンバイ中に、GNSS デバイスと SUPL スタックが NI 通知をウェイクアップできるようにする必要もあります。

    ドライバーの電源管理に関する一般的な情報については、「[電源管理の役割](https://docs.microsoft.com/windows-hardware/drivers/kernel/power-management-responsibilities-for-drivers)」を参照してください。

- **電源の考慮事項:** GNSS ドライバースタックでは、主な設計目標として電力のフットプリントを考慮し、メインプロセッサを可能な限りスリープ状態にする必要があります。 すべての高度な機能 (異なる種類の修正プログラムなど) を実行する必要があります。これは、メインアプリプロセッサが必要以上にアクティブにする必要がなく、ほとんどの処理をチップセット/低電力プロセッサにオフロードできます。 一般に、HLOS で特に指定されていない限り、GNSS ドライバーは常に最も重要な制約として電力消費を処理する必要があり、電力フットプリントを最小限に抑えて通常の操作を実行するように設計する必要があります。 GNSS driver インターフェイスは、モバイルデバイスが可能な限り低電力モードに移行できるように明示的に設計されています。また、必要な電源関連ヒントを GNSS ドライバーに提供して電力使用量を最適化することもできます。 追跡、ジオフェンシング、および長時間に実行される広範囲の使用を必要とするその他の機能については、GNSS ドライバー/エンジンが低電力のハードウェア/プロセッサを利用する必要があります。 このような機能を実装する必要がある場合は、ドライバーのブルートフォースポーリングメカニズムを使用するか、アプリプロセッサに実装する必要がある場合は、ドライバーがそのような操作に対応していることを宣言しないでください。 これにより、HLOS は、このような機能の公開をプラットフォームの残りの部分に制限したり、他のプラットフォームサービス/プリミティブに基づいて、その機能の代替実装を使用したりできます。

- **プログラミング言語:** GNSS driver インターフェイスは、 C++特定の言語プリミティブ (たとえば、構造の継承) を使用しない C 言語のヘッダーファイルとして配信されます。 これにより、IHV は C とC++のどちらかを選択できます。 GNSS インターフェイスヘッダーファイルは、選択したものを Ihv にオープンしたままにします。

- **フィルタードライバー:** GNSS デバイススタックは、拡張機能をサポートするために将来のフィルタードライバーをスタックに追加できないように制限することはできません。 IHV が提供する GNSS ドライバーは、ドライバースタックの上部または下部に独自のフィルタードライバーを含めることはできません。

- **通知とイベント:** WDF ドライバーは、要請されていない通知を HLOS に送信することができないため、常に、そのような要請されていない通知をドライバーレイヤーから受信することを目的とする、少なくとも1つの保留中の IRP を HLOS から開始します。 これには、システムがコネクトスタンバイ状態の場合も含まれます。 GNSS ドライバーの場合、このような要請されていない通知には、ネットワークによって開始された要求、AGNSS のサポートに関するサポートデータ、その他のベンダ固有の通知などが含まれます。 HLOS は、このような通知を処理するために、保留中の i/o 要求が常に存在することを保証します。

- **ユーザーモード IHV 拡張機能:** Ihv が定義したプライベート Ioctl を介して、GNSS ドライバーと対話するアクセサリユーザーモードコンポーネントを作成できます。 これは特に、GNSS ドライバーがカーネルモードの場合に必要です。この場合、ユーザーモード (Wi-fi スキャン、接続マネージャー Api など) でのみ使用できる機能にはアクセスできません。 Windows 10 の UMDF 2.0 では、UMDF GNSS ドライバーに個別のユーザーモードコンポーネントが必要ないことに注意してください。ただし、IHV は個別のユーザーモードコンポーネントを実装している可能性があります。 これらのユーザーモードコンポーネントは、GNSS ドライバーの単なる拡張機能として扱われ、IHV が提供する BSP ドロップの一部として扱われます。 Microsoft 提供の HLOS コンポーネントは、このようなコンポーネントの正確な実装の詳細と、IHV ユーザーモード/カーネルモードのコンポーネント間の相互作用メカニズムに無関係ます。 ユーザーモードの IHV 拡張機能を使用して、GNSS ドライバーが UMDF 2.0 ドライバーとして書き込まれている場合、このモデルではより多くのメモリ使用量が必要になる可能性があるため、推奨されません。

    ユーザーモード IHV 拡張機能は、次の規則に従っている必要があります。
    - パブリック GNSS ドライバーの Ioctl のセマンティクスと動作は、ユーザーモードの IHV 拡張機能と、GNSS ドライバーとの相互作用によって、影響を受けないままにしておく必要があります。
    - ユーザーモードの拡張機能は、Windows 10 プラットフォームによって課せられるセキュリティ、パワー、およびその他のプラットフォームの基本およびポリシーに準拠している必要があります。
    - ユーザーモード拡張機能は、Microsoft が承認した承認済みのアクティビティのみを実行する必要があります。 OS プラットフォームでは、実行時にこのような承認を強制/検証しません。

    > [!NOTE]
    > Microsoft でも、セキュリティポリシーを適用し、そのようなコンポーネントの有効期間を制御できます。 ここで重要な点は、IHV のユーザーモードコンポーネントは、拡張コンポーネントが信頼された OS コンポーネントとして扱われるため、このようなポリシーを適用するために、プラットフォームではカウントしないことです。

    Ihv は、任意の機能を追加したり、未承認の OS サービス/セキュリティで保護されたリソースを使用したりしません。
