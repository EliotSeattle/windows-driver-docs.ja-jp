# <a name="publishing-messages"></a>メッセージの公開


クライアントは、特定の種類のメッセージを公開できますが、メッセージを受信するユーザーや、メッセージが別のアプリまたはデバイスによって受信されたことについては、限られた知識を持つことになります。 これにより、クライアントは、メッセージ転送のタイミングを管理することなく、関心のあるユーザーに簡単に情報を "アドバタイズ" できます。

Windows 近接 API のクライアントは、メッセージの公開を要求できます。 パブリケーションの有効期間は、クライアントにバインドされます。 クライアントがを実行している限り、パブリケーションを維持できます。 クライアントは、いつでもパブリケーションを削除できます (非公開)。 クライアントが実行を停止すると、パブリケーションは削除されます。

メッセージが非公開になると、デバイスが将来近接されたときに送信されなくなります。

NFP モデルは、短いメッセージ (つまり、通常は512バイト未満) をサポートすることを目的としています。 NFP プロバイダーは、最大10キロバイト (KB) までのメッセージをサポートするためにのみ必要です。 すべてのプロバイダーで移植可能なコードを使用してサイズの大きいメッセージを送信するクライアントは、手動でメッセージをチャンクしたり、メッセージを帯域外に送信したりする必要があります。

## <a name="message-transmitted-callback"></a>メッセージ転送コールバック


メッセージを転送するコールバックによって、クライアントは、公開されたメッセージが別のデバイスに送信されたことを知ることができます。 メッセージ転送コールバックの目標の1つは、アプリが送信された後すぐにメッセージを非公開にできるようにすることです。

このモデルでは、受信側がメッセージを実際に処理することについての情報は提供されません。 これを実現する唯一の方法は、受信側がメッセージをクライアントに返信することです。
